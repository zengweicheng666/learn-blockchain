## 一、签名 vs 加密

上一节课说的是**加密**：用公钥锁住数据，只有私钥能打开。

这一节课说的是**签名**：用私钥打上一个"指纹"，任何有公钥的人都能验证这指纹确实是你的。

| 操作 | 使用 | 目的 |
|------|------|------|
| **加密** | 公钥加密 → 私钥解密 | 保密性——只有接收者能看 |
| **签名** | 私钥签名 → 公钥验签 | 认证性——证明是你发的 |

## 二、以太坊地址推导全流程

```
私钥 (32字节随机数)
    │
    ▼  secp256k1 标量乘法:  K = k * G
公钥 (65字节, 0x04 + x + y)
    │
    ▼ 去除 0x04 前缀
公钥原始字节 (64字节)
    │
    ▼ Keccak-256 哈希
哈希值 (32字节)
    │
    ▼ 取后 20 字节
地址 (0x + 40 个 hex 字符)
```

### 关键点

- 私钥 → 公钥：**单向推导**（ECDLP 保证不能反推）
- 公钥 → 地址：**单向哈希**（Keccak-256 保证不能反推）
- 地址 → 公钥：**不能**——但通过签名可以恢复公钥（`ecrecover`）

## 三、签名流程

```
消息 ──► Keccak-256 ──► 消息哈希 h
                                  │
私钥 d ──────────────────────────┤
随机数 k ──► R = k*G ──► r = R.x ┤
                                  ▼
                   s = k⁻¹ * (h + d * r) mod n
                                  ▼
                        签名 (v, r, s)
```

### 签名三个组成部分

| 字段 | 长度 | 含义 |
|------|------|------|
| r | 32 字节 | 随机点 R 的 x 坐标 |
| s | 32 字节 | 签名证明值 |
| v | 1 字节 | 恢复标识符（R 点的 y 奇偶性），27 或 28 |

### 为什么需要随机数 k？

> 如果两次签名用同一个 k，私钥可以被推算出来。2010 年索尼 PS3 的 ECDSA 签名实现用了固定的 k，导致私钥被破解。这就是为什么安全的随机数对签名至关重要。

## 四、验签流程

### 方式：ecrecover（以太坊做法）

不直接对比公钥，而是**从签名中恢复出签名者的地址**，然后和预期地址对比。

```
签名 (v, r, s) + 消息哈希 h
        │
        ▼  ecrecover 算法
    公钥 Q'  ──► Keccak-256 ──► 取后20字节 ──► 地址A'
        │                                           │
        └───────────── 是否相等？────────────────────┘
```

### 为什么 ecrecover 很强大？

- 验签**不需要**事先知道签名者的公钥
- 只需要签名和消息，就能算出"谁签的"
- 以太坊交易中的 `from` 地址其实不需要显式包含——从签名 recover 出来即可

## 疑问记录

1. 为什么以太坊用 Keccak-256 而不用 SHA256？——Keccak-256 是 SHA3 竞赛的胜出算法，以太坊在 SHA3 标准化前就采用了。标准 SHA3-256 其实和 Keccak-256 有细微差异，两者不兼容
2. v 值为什么老是 27 或 28？ ——因为早期比特币设计的约定，27 = 低 y 值 + 0x1b 恢复参数，后来以太坊沿用。EIP-155 之后 v = 2*chainId + 35/36 来防止跨链重放
