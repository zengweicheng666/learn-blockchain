## 一、节点类型对比

| 维度 | 全节点 | 轻节点 | 归档节点 |
|------|--------|--------|---------|
| 存储内容 | 完整区块头 + 所有交易 | 仅区块头 | 区块头 + 所有交易 + 全部历史状态 |
| 磁盘占用 | 以太坊 ~1TB，比特币 ~600GB | MB 级别 | 以太坊 ~14TB+ |
| 验证能力 | 完整验证每笔交易（签名、余额） | 仅用 Merkle Proof 验证交易存在性 | 可查询任意历史状态 |
| 运行成本 | 高（存储 + 带宽） | 极低 | 极高 |
| 典型用户 | 矿工、节点运营者 | 手机钱包、浏览器钱包 | Etherscan、Infura |

### 全节点（Full Node）

- 下载每个区块的**完整区块头 + 所有交易**
- 验证每一笔交易的合法性（签名、余额等）
- 代表：Geth 默认模式、Bitcoin Core
- 磁盘占用：以太坊 ~1TB+，比特币 ~600GB+

### 轻节点（Light Node）

- 只下载**区块头**（每个区块 ~80 字节 in 比特币，~500 字节 in 以太坊）
- 验证交易时：向全节点请求 Merkle Proof
- 代表：手机钱包（MetaMask Mobile）、浏览器轻钱包
- 磁盘占用：MB 级别

### 归档节点（Archive Node）

- 存储全部历史状态（每个区块的所有账户余额、合约存储等)
- 允许查询任意历史状态（如"某地址在 2023 年的余额"）
- 代表：Etherscan 的后端、Infura 归档节点
- 磁盘占用：以太坊 ~14TB+

## 二、轻节点的验证流程

```
轻节点（只存区块头）
    │
    │  我想要验证 tx5 是否在区块 #42 里
    ▼
全节点（存了全部交易）
    │
    │  返回 tx5 的 Merkle Proof（兄弟哈希列表）
    ▼
轻节点
    │
    │  用 Proof + 自己的 Merkle Root 逐步计算
    ▼
  根匹配？→ ✅ 交易存在 | ❌ 不存在或篡改
```

### 为什么轻节点不能验证"所有"规则？

- 它不知道账户余额，所以不能验证"发送方是否有足够余额"
- 它只能验证"交易是否在区块里"（存在性证明）
- 它必须信任全节点提供的区块头是正确的（除非自己偶尔连上全节点校验）

## 三、实操结果

> 运行 `npx tsx projects/merkle-tree/light-client-demo.ts`

```
场景：8 笔交易，轻节点只存 Merkle Root
验证 "eve -> frank 7.0 ETH"
Proof：4 个兄弟哈希，4 步计算得出 Root ✅

存储对比：
全节点:     ~180+ 字节（全部交易）
轻节点:      64 字节（仅 Root）
节省比例:   仅需 5% 的存储即可验证交易存在
```
